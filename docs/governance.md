# Governance — EventAI

## Risk Register

| # | Риск | Вероятность | Влияние | Детект | Защита | Остаточный риск |
|---|---|---|---|---|---|---|
| 1 | **LLM API недоступен** (OpenRouter down, rate limit, 429/503) | Средняя | Высокое — AI-фичи не работают | Мониторинг HTTP-статусов, fail_count в KeyManager | Ротация API-ключей (round-robin), cooldown 60 с, fallback-модель (GPT-4o-Mini), graceful degradation по уровням: рекомендации → tag overlap, профилирование → дефолтный вопрос, кластеризация → fail с уведомлением | Низкий: CRUD и навигация работают, AI-фичи деградируют но не ломают UX |
| 2 | **LLM генерирует невалидный JSON** при кластеризации (дубликаты, пропуски, «левые» id) | Высокая | Среднее — сломанное расписание | Валидатор: проверка полноты, уникальности, баланса после каждого LLM-вызова | Двухслойная валидация (синтаксис + семантика), безопасные автоисправления (missing → smallest room, duplicates → first), retry при синтаксических ошибках, аудит-трейл (source=llm/autofix/manual) | Низкий: ~3-5% проектов требуют ручной доводки, но ни один не теряется |
| 3 | **Prompt injection** через пользовательский текст в Telegram | Средняя | Среднее — агент может выполнить нежелательное действие | Логирование tool_calls и аргументов | Allowlist инструментов (7 функций), валидация аргументов по схеме (типы, обязательные поля), ограничение длины входного текста | Средний: нет runtime sandbox, PII-фильтр отсутствует |
| 4 | **Prompt injection** через описания проектов | Средняя | Среднее — влияние на рекомендации/кластеризацию | Ручная проверка организатором при импорте | В LLM отправляется только title/tags/description (до 500 символов), без авторов/контактов. Кластеризация модерируется организатором | Средний: автоматической фильтрации нет |
| 5 | **Утечка персональных данных** во внешний LLM API | Низкая | Высокое — нарушение 152-ФЗ | Аудит отправляемых полей в логах | Telegram username и ФИО **не** отправляются в LLM. Из проектов — только title/tags/description. Из экспертов — только теги. OpenRouter как прокси (данные не хранятся на стороне провайдера) | Средний: нет автоматического PII-детектора, raw_text пользователя может содержать PII |
| 6 | **Потеря сессии** при перезапуске контейнера | Средняя | Низкое — пользователь теряет прогресс диалога | Мониторинг uptime контейнеров | PicklePersistence для сохранения state между перезагрузками. Orphan handler восстанавливает пользователя из БД по /start. Профиль и рекомендации хранятся в PostgreSQL (не теряются) | Низкий: теряется только in-progress диалог профилирования |
| 7 | **Перегрузка в день DD** (200 одновременных сессий) | Средняя | Высокое — бот тормозит/не отвечает | Мониторинг RPS, queue length в Celery/RabbitMQ | Celery workers для async LLM-операций, typing-индикатор при ожидании >1с, LLM-кэш для Q&A-вопросов (TTL 1 час), batch processing для уведомлений | Средний: нет rate limiting, нет автоскейлинга |
| 8 | **Некорректные рекомендации** (bias, нерелевантные проекты) | Средняя | Низкое — пользователь получает неоптимальную программу | Офлайн-eval (Precision/Recall/NDCG на 10 профилях) | Пользователь может rebuild_profile, compare_projects для ручной проверки. Fallback на tag overlap при сбое embeddings | Средний: нет online-метрик, нет baseline-сравнения, разметка без IAA |
| 9 | **Qdrant недоступен** | Низкая | Среднее — рекомендации деградируют | Health-check Qdrant в Docker | Автоматический fallback на tag overlap scoring (overlap * 20.0). Качество хуже, но рекомендации не «умирают» | Низкий: пользователь всё равно получает результат |
| 10 | **Злоупотребление ботом** (спам, DoS, троллинг) | Низкая | Среднее — расход токенов LLM, деградация для других пользователей | Нет автоматического детекта | **Не реализовано.** Rate limiting отсутствует. Открытая задача | Высокий: бот открыт для любого пользователя Telegram |

## Политика логов и работы с персональными данными

### Классификация данных

| Категория | Данные | Уровень |
|---|---|---|
| Идентификаторы | Telegram ID (числовой), internal UUID | Псевдонимизированные |
| Персональные | ФИО, Telegram username | Персональные (152-ФЗ) |
| Профиль интересов | Теги, ключевые слова, цели, компания | Персональные (152-ФЗ) |
| Сырой текст | raw_text диалога профилирования | Может содержать PII |
| Проектные данные | Названия, описания, теги проектов | Общедоступные |
| Оценки экспертов | Баллы по критериям | Конфиденциальные |

### Что хранится в БД (PostgreSQL)

- Полный профиль пользователя: selected_tags, keywords, raw_text, extra_data (company, position, nl_summary)
- История рекомендаций: project_id, relevance_score, llm_summary
- История кластеризаций: raw_prompt, raw_response, model, prompt_hash, input_hash
- Бизнес-followup: статусы проектов, LOI

### Что отправляется во внешний LLM (OpenRouter API)

| Отправляется | НЕ отправляется |
|---|---|
| Интересы, цели, роль гостя | Telegram username, ФИО |
| Компания, должность (для бизнеса) | Telegram ID |
| title, tags, description проектов (до 500 символов) | Авторы, контакты проектов |
| Теги экспертов (для adjacent tags) | Имена, контакты экспертов |
| Chat history (до 20 сообщений) | Оценки экспертов |

### Retention

Формальная политика retention **не определена** (открытая задача для продакшена). Текущее состояние: данные хранятся бессрочно в PostgreSQL.

### Логирование

- Логи **не содержат** персональных данных (ФИО, контакты)
- Telegram ID допустим как псевдоним в логах
- LLM-вызовы логируются: model, tokens_in/out, latency, key_id (без содержимого промптов в production-логах)
- Кластеризация: prompt_hash и input_hash для воспроизводимости (raw_prompt/raw_response — в БД, не в логах)

### Локализация данных

- Все данные хранятся в РФ (Yandex Cloud, ЦОД в России) — требование 152-ФЗ
- OpenRouter API — внешний сервис (данные покидают РФ при LLM-вызовах). Отправляемые данные минимизированы (см. таблицу выше)

## Защита от injection и подтверждение действий

### Prompt Injection

**Два вектора атаки:**

1. **Пользовательский текст в Telegram** — пользователь может написать «игнорируй предыдущие инструкции» или вставить инструкции в свободный текст профилирования
2. **Описания проектов** — при импорте из Excel описания пишут разные люди; теоретически можно вложить инструкции для модели

**Реализованные защиты:**

| Мера | Реализация |
|---|---|
| Allowlist инструментов | Агент может вызвать только 7 заранее описанных функций: show_project, show_profile, compare_projects, generate_questions, get_followup, get_pipeline, rebuild_profile |
| Валидация аргументов | Каждый tool_call проверяется по типам и обязательным полям перед выполнением |
| Изоляция контекстов | Контакты и персональные данные организатора не прокидываются в промпты рекомендаций/кластеризации |
| Ограничение данных проектов | В LLM для проектов — только title/tags/description с ограничением длины; авторы и контакты не отправляются |
| Ограничение данных пользователя | Telegram username и ФИО не отправляются в LLM |
| Модерация AI-решений | Кластеризация, обратная связь — утверждаются организатором через админку |

**Не реализовано (открытые задачи):**

| Мера | Статус |
|---|---|
| Автоматическая PII-фильтрация перед отправкой в LLM | Не реализовано |
| Rate limiting на входе Telegram-бота | Не реализовано |
| JSON Schema валидация ответов кластеризации (strict mode) | Не реализовано (используется post-hoc валидатор) |
| Sandbox для выполнения tool_calls | Не реализовано (tools выполняются в том же процессе) |
| Content filtering входящих описаний проектов | Не реализовано |

### Подтверждение действий

Все критические AI-решения проходят через человека:

| Действие | Кто подтверждает | Как |
|---|---|---|
| Кластеризация проектов по залам | Организатор | Просмотр результата в UI → drag-n-drop правки → кнопка «Подтвердить» |
| Распределение экспертов | Организатор | Просмотр матчинга → ручные корректировки → подтверждение |
| Расписание | Организатор | Просмотр → drag-n-drop → экспорт |
| Рассылки/брифинги | Организатор | Превью текста → подтверждение отправки |
| Профиль гостя | Гость | «Всё верно» / «Ввести заново» перед генерацией рекомендаций |
| Рекомендации | Гость | Может rebuild_profile и перегенерировать |
