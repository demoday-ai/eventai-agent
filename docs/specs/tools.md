# Tools - Спецификация

7 инструментов агента в режиме VIEW_PROGRAM. Определены в `AGENT_TOOLS` (`start.py`), диспетчеризация в `view_program_text()`.

---

## Общая архитектура

1. Пользователь пишет сообщение в чате
2. Система строит system prompt + chat history + tools schema
3. LLM возвращает `{"type": "tool_call", "tool_name": "...", "tool_args": {...}}` или `{"type": "text", "content": "..."}`
4. Бот диспетчеризирует вызов инструмента, возвращает результат пользователю

---

## 1. rebuild_profile

Перезапуск профилирования - пользователь хочет изменить интересы.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | Нет |
| Выходной формат | Переход в состояние NL_REBUILD |
| Таймаут | Нет (мгновенный переход состояния) |
| Side effects | Очистка `program_chat`, `recommendations` из user_data. Сброс `nl_conversation` |
| Обработка ошибок | Если нет роли -> "Сначала выберите роль через /start" |
| Доступ | Все роли (guest, business) |

Поведение: вызывает `_restart_nl_profiling()`, возвращает состояние ONBOARD_NL_PROFILE (NL_REBUILD). После подтверждения нового профиля -> повторная генерация рекомендаций.

---

## 2. show_project

Показать детали конкретного проекта по номеру в рекомендациях.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | `project_rank` (integer) - номер проекта в списке (1, 2, 3...) |
| Выходной формат | Карточка проекта: title, LLM-summary или description[:300], tags, room, relevance score |
| Таймаут | Нет (DB query) |
| Side effects | Сохраняет `current_project_id`, `current_project_title`, `current_project_rank` в user_data |
| Обработка ошибок | Если rank не найден -> "Проект #{rank} не найден в рекомендациях" |
| Доступ | Все роли |

Поведение: находит project_id по rank в списке рекомендаций (must_visit + if_time), загружает полные данные из DB через `get_project_detail()`, переводит в состояние VIEW_DETAIL с кнопками "Подготовить вопросы", "Запросить контакт", "Назад".

---

## 3. show_profile

Показать текущий профиль пользователя.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | Нет |
| Выходной формат | Текстовое сообщение: теги, ключевые слова, NL-описание, компания (для бизнеса), бизнес-цели |
| Таймаут | Нет (in-memory data) |
| Side effects | Нет |
| Обработка ошибок | Если profile_info пуст -> "Профиль пуст" |
| Доступ | Все роли |

Поведение: возвращает строку `profile_info`, собранную из user_data. Не требует DB-запроса (данные уже загружены в контексте агента).

---

## 4. compare_projects

Сравнить несколько проектов - матрица сравнения по критериям.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | `project_ranks` (array of int, 2-5 элементов), `criteria` (array of string, опционально) |
| Выходной формат | Текстовая матрица: критерий x проект |
| Таймаут | 25 секунд (wait_for_task) |
| Side effects | Создает Celery-задачу `generate_comparison_matrix_task` |
| Обработка ошибок | <2 проектов -> "Для сравнения нужно минимум 2 проекта". >5 -> обрезается до 5. Timeout -> "Не удалось сгенерировать сравнение" |
| Доступ | Все роли |

Поведение:
1. Маппинг ranks -> project_ids из рекомендаций
2. Если criteria не указаны -> `get_default_criteria()` (учитывает бизнес-профиль)
3. Celery task: LLM генерирует матрицу
4. Форматирование через `format_matrix_text()`
5. Отправка через `safe_send_long_message()` (разбиение длинных сообщений)

---

## 5. generate_questions

Подготовить вопросы для Q&A к конкретному проекту.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | `project_rank` (integer) |
| Выходной формат | Нумерованный список 3-5 вопросов |
| Таймаут | 20 секунд (wait_for_task) |
| Side effects | Создает Celery-задачу `generate_qa_questions_task` |
| Обработка ошибок | Проект не найден -> "Проект #{rank} не найден". Timeout -> "Не удалось сгенерировать вопросы" |
| Доступ | Все роли |

Поведение:
1. Маппинг rank -> project_id
2. Celery task: загрузка user + project + guest_profile + business_profile из DB
3. LLM генерирует контекстно-зависимые вопросы (учитывает профиль пользователя)
4. Формат: `*Вопросы для проекта #{rank}:*\n_{title}_\n\n1. ...\n2. ...`

## 6. get_followup

Follow-up пакет после Demo Day: список посещенных проектов, контакты авторов, шаблон письма.

| Параметр | Значение |
|----------|---------|
| Входные аргументы | Нет |
| Выходной формат | Форматированное сообщение с проектами, контактами, шаблоном email |
| Таймаут | DB queries (секунды) |
| Side effects | Создает follow-up package в DB при первом вызове (`get_or_create_package()`) |
| Обработка ошибок | Бизнес -> "Follow-up пакет доступен только для гостей". Нет данных -> "Пока нет данных для follow-up" |
| Доступ | Только guest (роль проверяется в коде) |

Поведение:
1. Проверка роли: business -> отказ с подсказкой get_pipeline
2. Загрузка/создание follow-up package
3. Форматирование через `format_package_message()`
4. Отправка через `safe_send_long_message()`

---

## 7. get_pipeline

Бизнес-пайплайн: проекты по статусам (заинтересован, на связи, переговоры, сделка).

| Параметр | Значение |
|----------|---------|
| Входные аргументы | Нет |
| Выходной формат | Форматированное сообщение с проектами, сгруппированными по статусам |
| Таймаут | DB queries (секунды) |
| Side effects | При пустом pipeline -> `init_pipeline_from_recommendations()` (создание из рекомендаций) |
| Обработка ошибок | Guest -> "Пайплайн доступен только для бизнес-партнеров". Пуст -> "Пайплайн пуст" |
| Доступ | Только business (роль проверяется в коде) |

Поведение:
1. Проверка роли: guest -> отказ с подсказкой get_followup
2. Загрузка pipeline projects из DB
3. Если пуст -> автоматическая инициализация из рекомендаций
4. Загрузка бизнес-профиля для контекста
5. Форматирование через `format_pipeline_message()`
6. Отправка через `safe_send_long_message()`

## Планируемые инструменты

### ask_organizer

Маршрутизация вопроса пользователя к организатору через support chat.

| Параметр | Значение |
|----------|---------|
| Статус | Планируется |
| Текущее решение | Отдельное состояние SUPPORT_CHAT (не через tool calling) |
| Входные аргументы | Текст вопроса |
| Поведение | Создание thread в support system, пересылка в админку, ожидание ответа |

## Защита и валидация

### Tool whitelist

Агент может вызвать только инструменты из `AGENT_TOOLS` - 7 функций, определенных явно. LLM не может вызвать произвольную функцию. Неизвестный tool_name -> "Неизвестное действие".

### Валидация аргументов

- `project_rank` - проверка типа (integer), поиск в списке рекомендаций
- `project_ranks` - проверка минимум 2, максимум 5 (обрезается)
- `criteria` - опциональный, при отсутствии генерируются автоматически

### Контроль доступа по ролям

| Инструмент | guest | business |
|-----------|-------|----------|
| rebuild_profile | + | + |
| show_project | + | + |
| show_profile | + | + |
| compare_projects | + | + |
| generate_questions | + | + |
| get_followup | + | - (перенаправляет на get_pipeline) |
| get_pipeline | - (перенаправляет на get_followup) | + |

Проверка роли выполняется в обработчике каждого инструмента через `context.user_data["pending_role_code"]`.

### Парсинг LLM-ответа

LLM-ответ парсится через `send_chat_with_tools()`:
1. Проверка наличия `tool_calls` в response message
2. Парсинг `function.arguments` как JSON
3. Диспетчеризация по `tool_name` в `view_program_text()`
4. Если `tool_name` не в whitelist -> "Неизвестное действие"
5. Если `type == "text"` и content пуст -> "Не удалось получить ответ"

### Обработка ошибок

```
tool dispatch exception -> "Произошла ошибка. Попробуйте ещё раз или используйте кнопки."
Celery timeout         -> "Не удалось сгенерировать X. Попробуйте позже."
LLM timeout           -> "Обработка занимает больше времени. Попробуйте ещё раз."
```

Все ошибки логируются через `logger.exception()`. Агент не прерывает сессию при ошибке инструмента - остается в VIEW_PROGRAM.
